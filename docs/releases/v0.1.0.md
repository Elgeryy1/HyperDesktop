# HyperDesk v0.1.0

Fecha: 2026-03-26
Tag: `v0.1.0`

## Resumen rapido

`v0.1.0` es la primera release publica de HyperDesk.
Entrega un MVP funcional para administrar maquinas virtuales desde una interfaz web, con autenticacion, roles, auditoria y operaciones de consola remota.

## Que es HyperDesk

HyperDesk es una plataforma web para operar infraestructura de VMs tipo "cloud lab" desde un panel central.
Su objetivo en esta version es cubrir el flujo operativo base:

- Gestionar usuarios y permisos.
- Registrar hosts de virtualizacion.
- Crear y operar VMs.
- Cargar ISOs y definir templates.
- Abrir consola remota VNC/RDP.
- Auditar operaciones criticas.

## Como funciona (vision tecnica)

La arquitectura es un `modular monolith` en monorepo:

- `apps/web`: Next.js (interfaz web).
- `apps/api`: Express + TypeScript (API y logica de negocio).
- `PostgreSQL + Prisma`: persistencia.
- `Docker Compose`: despliegue local.

La capa de virtualizacion se desacopla via provider:

- `mock`: para desarrollo/demo sin hypervisor real.
- `libvirt`: para operaciones reales via `virsh`.

## Flujo operativo principal

1. El usuario inicia sesion en la web.
2. La API valida credenciales y emite `access token` + `refresh token`.
3. El frontend consulta recursos segun permisos (`ADMIN`, `OPERATOR`, `VIEWER`).
4. Las acciones sobre VM (create/start/stop/reboot/delete) se validan en API.
5. La API ejecuta la accion en el provider seleccionado (`mock` o `libvirt`).
6. Cada accion critica queda registrada en `audit-logs`.
7. Para consola remota, la API crea una sesion temporal y devuelve `launchUrl` (VNC o RDP).

## Funcionalidad incluida en v0.1.0

### Seguridad y acceso

- Login, refresh y logout.
- JWT + sesiones de refresh rotatorio.
- RBAC basico con 3 roles.

### Operacion de plataforma

- CRUD de usuarios.
- Gestion de roles.
- Gestion de grupos y miembros.
- Dashboard de estado y recursos.

### Recursos de virtualizacion

- Hypervisors con alta, edicion y baja.
- `Probe` de hypervisor para sincronizar CPU/RAM, redes y pools.
- Virtual Machines con listado, detalle, creacion, edicion y borrado.
- Acciones de ciclo de vida de VM: start, stop y reboot.
- Scheduler base para acciones programadas de VM.
- Storage pools y volumenes.
- Networks con listado, alta y actualizacion.
- ISOs con carga por `multipart`, registro y eliminacion.
- Templates con alta, listado y borrado.

### Consola remota

- VNC web via noVNC.
- RDP via archivo `.rdp`.
- Tunel RDP automatico en modo `libvirt` (`REMOTE_CONSOLE_RDP_TUNNEL_MODE=libvirt_ssh`).

### Observabilidad funcional

- Auditoria de eventos criticos.

## Como usar HyperDesk (paso a paso)

### 1) Requisitos

- Docker + Docker Compose.
- Node.js 22+.
- npm 11+.

### 2) Arranque recomendado

En la raiz del proyecto:

```powershell
.\start-hyperdesk.ps1
```

Ese script:

1. Crea `.env` desde `.env.example` si no existe.
2. Levanta el stack con Docker Compose.
3. Espera a que PostgreSQL este healthy.
4. Ejecuta migraciones Prisma.
5. Ejecuta seed inicial.

### 3) URLs de acceso

- Web: `http://localhost:3000`
- API: `http://localhost:4000`
- Health: `http://localhost:4000/health`
- noVNC (mock): `http://localhost:6081`

### 4) Credenciales iniciales

- Email: `admin@hyperdesk.local`
- Password: `ChangeMe123!`

Recomendacion: cambiar password en cuanto inicies sesion.

### 5) Flujo de uso sugerido

1. Inicia sesion con el admin inicial.
2. Revisa `Hypervisors` y confirma host activo.
3. Carga una ISO en `ISOs` (si vas a crear VM desde instalador).
4. Verifica/redacta `Networks` y `Storage`.
5. Crea una VM desde `Virtual Machines`.
6. Ejecuta start/reboot/stop segun pruebas.
7. Abre consola `VNC` o descarga archivo `RDP`.
8. Revisa `Audit Logs` para trazabilidad.

## Modo mock vs modo libvirt

### Modo mock (default para local)

- `HYPERVISOR_PROVIDER=mock`
- Ideal para pruebas funcionales de UI/API sin hypervisor real.

### Modo libvirt (real)

1. Ajustar `.env`:

```env
HYPERVISOR_PROVIDER=libvirt
LIBVIRT_DEFAULT_URI=qemu:///system
LIBVIRT_STORAGE_POOL=default
LIBVIRT_DEFAULT_NETWORK=default
```

2. Reiniciar stack:

```powershell
.\stop-hyperdesk.ps1
.\start-hyperdesk.ps1
```

3. Ejecutar `Probe` en la UI para sincronizar recursos reales.

## API principal (resumen)

Base URL: `/api/v1`

- `auth`: login, refresh, logout, me.
- `users`: CRUD de usuarios.
- `roles`: listar/actualizar permisos por rol.
- `virtual-machines`: CRUD + acciones start/stop/reboot.
- `hypervisors`: CRUD + probe.
- `groups`: CRUD de grupos + miembros.
- `storage`: pools y volumenes.
- `isos`: listar, registrar, upload, delete.
- `templates`: listar, crear, delete.
- `networks`: listar, crear, editar.
- `remote-console`: sesiones VNC/RDP.
- `dashboard`: resumen y recursos.
- `audit-logs`: trazabilidad.
- `settings`: leer/actualizar configuracion.

## Limitaciones actuales (v0.1.0)

- Sin HA de control-plane.
- Sin quotas avanzadas multi-tenant.
- Sin hardening completo para produccion.
- Sin pipeline CI/CD y observabilidad full.

## Recomendaciones para despliegue serio

1. Rotar secretos JWT y credenciales admin.
2. Configurar HTTPS reverse proxy.
3. Restringir CORS y puertos expuestos.
4. Aplicar backup/restore para Postgres.
5. Definir monitoreo y alertas.
